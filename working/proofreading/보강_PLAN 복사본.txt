# 교과서·참고서 교정/교열 지원 서비스 기획 초안 보강안

## Summary
- 현재 플랜의 방향은 타당합니다. 특히 `자동 수정`이 아니라 `추천 + 사람 검토`로 시작하는 점, 그리고 `도서 내 용어 일관성`을 핵심 차별점으로 잡은 점이 현실적입니다.
- 다만 구현 전에 더 선명하게 고정해야 할 부분은 `MVP 경계`, `사용자 역할`, `문서 단위`, `검사 결과 반영 방식`, `HWP 처리 한계`입니다.
- 이번 보강안은 저장용 기획 초안에 가깝게, 서비스 범위를 흔들리지 않게 만드는 데 초점을 둡니다.

## Key Changes
- 문제 정의를 더 명확히 고정
  - 이 서비스의 1차 목적은 “교정자가 처음부터 끝까지 직접 찾는 반복적 오류를 먼저 수집해 검토 시간을 줄이는 것”입니다.
  - 1차 대상은 `출판사 내부 편집자/교정자`이며, 저자용 공개 SaaS는 범위 밖으로 둡니다.
- MVP 범위를 더 좁혀서 결정
  - 입력 단위는 `도서 전체`가 아니라 `챕터/단원 단위 문서`를 기본으로 합니다.
  - 업로드 즉시 완전 자동 처리보다, `문서 등록 -> 검사 실행 -> 결과 검토`의 3단계 흐름으로 둡니다.
  - 결과 반영은 원문 직접 수정이 아니라 `추천 목록과 판정 이력 관리`까지만 포함합니다.
  - 출력은 수정된 원고 재생성보다 `검토 리포트 다운로드`를 우선합니다.
- 핵심 기능 우선순위를 고정
  - 1순위: 맞춤법, 띄어쓰기, 문장부호, 숫자/단위 표기
  - 2순위: 용어/표기 일관성, 제목 스타일 일관성
  - 3순위: 반복 표현, 어색한 문장 경고
  - 초기에 문체 개선이나 의미 기반 재작성 제안은 제외합니다.
- 사용자 역할을 최소 단위로 정의
  - `검수자`: 오류 후보를 승인/반려/보류
  - `관리자`: 문서 등록, 검사 실행, 용어집 관리
  - 저자/외부 협업자는 후속 단계로 미룹니다.
- 기술/운영 리스크를 플랜에 반영
  - `HWP`는 초기에 정확도 편차가 클 수 있으므로 “지원은 하되 추출 품질 예외가 있음”을 전제로 설계합니다.
  - 검사 엔진은 하나의 AI 호출에 의존하지 않고 `규칙 기반 검사 + LLM 보조 설명/추천` 구조를 기본안으로 둡니다.
  - 문서 위치 추적은 페이지 번호보다 `문단/문장/토큰 인덱스` 기준으로 저장합니다.

## Public Interfaces / Data Shape
- `Document`
  - 문서 메타데이터, 도서명, 챕터명, 파일 형식, 업로드 버전, 검사 상태
- `Issue`
  - 오류 유형, 심각도, 원문 위치, 원문 일부, 탐지 근거, 묶음 키
- `Suggestion`
  - 추천 문구, 설명, 규칙 출처, 신뢰도
- `ReviewDecision`
  - 승인/반려/보류, 처리자, 처리 시각, 메모
- `GlossaryTerm`
  - 표준 용어, 허용 표기, 금지 표기, 적용 범위(도서/시리즈)

## Test Plan
- 범위 검증
  - 챕터 단위 문서 기준으로 업로드-검사-검토 흐름이 끊기지 않는지
  - 원문 직접 수정 없이도 실무 검토에 충분한 정보가 보이는지
- 품질 검증
  - 대표적인 맞춤법/띄어쓰기/단위 표기 사례를 안정적으로 검출하는지
  - 동일 개념의 서로 다른 표기가 한 묶음 이슈로 제시되는지
  - 제목/목차 스타일 불일치를 구조적으로 잡는지
- 운영 검증
  - HWP 추출 실패 시 사용자에게 실패 원인과 대안을 안내하는지
  - 승인/반려/보류 이력이 남고 재검사 후에도 비교 가능한지
- 수용 기준
  - 검수자가 “읽으며 찾기”보다 “추천 목록 검토”로 일하는 비중이 커져야 함
  - 문서 하나 기준으로 검토 시작까지의 준비 시간이 짧아야 함
  - 거짓 양성은 있더라도, 왜 추천됐는지 근거가 보여야 함

## Assumptions
- 지금 단계에서는 파일 저장이나 구현은 하지 않고, 저장 가능한 문서 초안을 다듬는 데 집중합니다.
- 저장 시 문서명은 `service-plan.md` 같은 기획 초안 파일로 두는 것이 가장 무난합니다.
- 다음 보강 라운드에서는 아래 셋 중 하나를 깊게 파는 것이 좋습니다.
  - 사용자 화면 흐름
  - 시스템 아키텍처
  - 사업성/운영 정책
