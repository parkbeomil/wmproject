# 교과서·참고서 교정/교열 지원 서비스 MVP 계획

## Summary
- 목표는 `DOCX/HWP 원고 업로드 -> 자동 교정 추천 -> 편집자 검토/반영 -> 이력 추적` 흐름을 웹 서비스로 제공하는 것입니다.
- MVP는 `AI가 자동 수정하지 않고 추천만 제시`하며, `국립국어원/일반 규범 기반 검사`와 `도서 내 용어·표기 일관성 검출`을 핵심 가치로 둡니다.
- 1차 성공 기준은 “사람이 처음부터 끝까지 읽으며 찾던 반복 오류를 서비스가 먼저 추려줘서, 교정 시간이 체감상 줄어드는 것”입니다.

## Key Changes
- 사용자 흐름
  - 원고 파일(`.docx`, `.hwp`) 업로드
  - 본문 텍스트/문단/표 제목/주석 추출
  - 문장 단위 검사 수행
  - 오류 유형별 추천 목록 제공
  - 편집자가 항목별 `승인 / 반려 / 보류` 처리
  - 반영 이력과 변경 상태 저장
- MVP 기능 범위
  - 맞춤법, 띄어쓰기, 문장부호, 반복 표현, 숫자/단위 표기, 용어 불일치 검사
  - 같은 도서 내 동일 개념의 다른 표기 탐지
  - 장/절/단원 단위 비교로 제목 스타일 불일치 탐지
  - 검수 상태 대시보드: 전체 오류 수, 미처리 항목, 반영률
  - 검토 결과 다운로드: 교정 제안 목록과 처리 상태
- 제외 범위
  - PDF 정밀 교정
  - InDesign 직접 연동
  - AI 자동 반영
  - 과목별 복잡한 규칙엔진
- 시스템 구성
  - 프론트엔드: 웹 업로드, 검수 화면, 변경 이력 화면
  - 백엔드 API: 파일 관리, 텍스트 추출, 검사 실행, 결과 저장, 검수 상태 관리
  - 검사 엔진: 규범 검사 + 일관성 검사 + 용어 사전 비교
  - 저장소: 문서 메타데이터, 추출 텍스트, 검사 결과, 사용자 판정 이력
- 인터페이스/데이터 모델
  - `Document`: 도서/챕터 단위 원고
  - `Issue`: 오류 후보, 위치, 유형, 심각도, 근거
  - `Suggestion`: 추천 수정안, 설명, 규칙 출처
  - `ReviewDecision`: 승인/반려/보류, 처리자, 처리 시각
  - `GlossaryTerm`: 표준 용어와 허용 변형
- 구현 기본 방침
  - HWP는 원문 바이너리를 직접 다루기보다 텍스트 추출 파이프라인을 별도 모듈로 분리
  - 검사 결과는 원문을 직접 수정하지 않고 `원문 span + 제안` 구조로 저장
  - 규범 검사는 범용 언어 규칙을 먼저 적용하고, 일관성 검사는 문서 전체를 한 번 더 스캔하는 2단계로 처리
  - 모델 출력은 항상 사람이 검수할 수 있게 근거와 함께 표시

## Test Plan
- 파일 처리
  - DOCX 업로드 시 문단/제목/표 캡션이 올바르게 추출되는지
  - HWP 업로드 시 추출 실패 문서를 사용자에게 명확히 안내하는지
- 검사 품질
  - 맞춤법/띄어쓰기/문장부호 대표 사례를 올바르게 검출하는지
  - 같은 개념의 표기가 문서 내에서 다를 때 일관성 이슈로 묶이는지
  - 숫자, 단위, 괄호, 따옴표 스타일 불일치를 잡는지
- 검수 흐름
  - 편집자가 승인/반려/보류한 상태가 저장되고 다시 열어도 유지되는지
  - 여러 사용자가 같은 문서를 검토할 때 이력 충돌 없이 상태가 보이는지
- 결과 제공
  - 필터(오류 유형, 미처리, 장/절 기준)가 정확히 동작하는지
  - 다운로드 결과에 원문 위치, 제안, 판정 상태가 포함되는지

## Assumptions
- 초기 사용자는 출판사 내부 편집자/교정자이며, 외부 공개 서비스보다 사내 업무 도구 성격이 강합니다.
- MVP의 핵심은 “정답을 자동 확정”하는 것이 아니라 “검토할 후보를 빠르게 모아주는 것”입니다.
- 규칙 기준은 우선 일반 국어 규범으로 시작하고, 사내 스타일가이드/용어집은 2차 단계에서 추가합니다.
- 기술 스택은 `웹 프론트엔드 + 백엔드 API + 문서 추출 모듈 + 검사 엔진` 구조를 기본으로 잡습니다.
- 다음 단계에서는 1) 사내 용어집 업로드, 2) 과목별 규칙셋, 3) PDF 비교/교정, 4) 기존 제작 시스템 연동 순으로 확장합니다.
